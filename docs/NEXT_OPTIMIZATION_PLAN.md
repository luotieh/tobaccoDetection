# AI 烟草违法监测系统：下一步优化实施任务书

> 用途：供后续 Codex / AI 编程助手继续生成、拆解和实现项目优化。  
> 当前基线：管理端已接入视觉、文本、语音三类识别服务，并具备识别内容列表、规则词库、审核管理、推送管理和前端测试入口。  
> 编写目标：把下一步优化方向整理为可落地的工程任务，避免后续生成时重新判断架构边界。

---

## 1. 当前基线

### 1.1 已完成能力

当前项目已经形成一个可运行的多模态违法烟草交易监测原型：

1. 管理端
   - 入口文件：`app.py`
   - 前端文件：`static/app.js`、`static/style.css`
   - 数据存储：SQLite `data/demo.db`
   - 主要页面：
     - 工作台
     - 识别内容列表
     - 图像识别测试
     - 文本识别测试
     - 语音识别测试
     - 模型配置
     - 多模态融合配置
     - 规则词库
     - 审核管理
     - 推送管理

2. 视觉识别服务
   - 目录：`app/`
   - 默认端口：`9000`
   - 接口：
     - `GET /health`
     - `GET /models/info`
     - `POST /infer/image`
     - `POST /infer/video`
   - 支持 YOLO 权重推理和 Mock 模式。
   - 管理端图像测试页支持 `basant-yolo26s` 和 `enos-yolo11m` 模型切换。

3. 文本识别服务
   - 目录：`text_service/`
   - 默认端口：`8010`
   - 接口：
     - `GET /health`
     - `GET /models/info`
     - `POST /infer/text`
     - `POST /infer/content`
   - 已支持规则词库、关键词命中、品牌词、联系方式脱敏、风险评分和解释生成。

4. 语音识别服务
   - 目录：`audio_service/`
   - 默认端口：`8020`
   - 接口：
     - `GET /health`
     - `GET /models/info`
     - `POST /infer/audio`
     - `POST /infer/video-audio`
   - 已支持音频/视频上传、音频抽取、Mock ASR、Whisper ASR 适配、关键词识别、证据片段和风险评分。

5. 多模态融合
   - 管理端识别内容时会根据内容类型调用文本、图像、语音服务。
   - 融合评分会综合文本、图像、语音和账号行为分。
   - 中高风险内容进入待审核流程。

### 1.2 当前主要问题

1. 运行模式容易混淆
   - 视觉服务、语音服务均支持 Mock/Test/真实模型模式。
   - 当前 README 中有启动示例，但缺少统一的 profile 定义和一键编排。
   - 演示时容易误把 Mock 命中当成真实识别命中。

2. 词库来源分散
   - 管理端 SQLite 中有 `rule_words`。
   - 文本服务和语音服务各自有 JSON 词库。
   - 目前存在部分同步和兜底逻辑，但没有形成统一的词库发布机制。

3. 测试数据仍偏运行态
   - 多模态命中样例可通过数据库生成，但没有固化为可重复的 fixtures。
   - 图像、音频、视频命中样本缺少统一存放、说明和验收脚本。

4. 业务闭环仍偏演示
   - 审核、推送、推送失败重试仍以 Mock 流程为主。
   - 缺少推送载荷规范、外部系统错误码、审计日志和操作追踪。

5. 服务工程化不足
   - `docker-compose.yml` 只编排文本和语音服务，未覆盖管理端和视觉服务。
   - 没有统一 stop/restart/status 脚本。
   - 服务健康检查没有在管理端形成完整的运维视图。

---

## 2. 优化目标

下一阶段目标不是继续堆演示功能，而是把现有原型推进到“可稳定演示、可重复测试、可继续生产化”的状态。

优先级排序如下：

1. 先保证运行和演示稳定。
2. 再保证规则、样本、模型结果可解释且可复现。
3. 再增强管理端业务闭环。
4. 最后补齐生产化能力。

---

## 3. 分阶段实施任务

### 阶段 1：运行与演示稳定化

#### 任务 1.1：统一运行模式

目标：明确 dev、demo、real 三种运行模式，避免 Mock/Test/真实模型混用。

当前问题：

- README 中有多个独立启动命令，但没有统一模式定义。
- 语音 Mock 转写、视觉 Mock 检测和真实模型推理之间容易混淆。

实施内容：

1. 新增或整理 `.env.example`，覆盖管理端、视觉、文本、语音所有关键环境变量。
2. 定义三种 profile：
   - `dev`：服务可启动，允许 Mock，适合开发联调。
   - `demo`：使用固定测试样本，允许显式 Mock 命中，适合现场演示。
   - `real`：关闭 Mock fallback，要求真实 YOLO 权重和真实 ASR 可用。
3. README 增加“运行模式说明”，明确每种模式的启动命令、预期结果和限制。

涉及模块：

- `README.md`
- `.env.example`
- `scripts/`

验收标准：

- 开发者能根据一个模式名称启动对应服务。
- `demo` 模式结果明确标注为演示结果。
- `real` 模式缺少模型或 ASR 时直接报错，不静默返回 Mock 命中。

推荐测试：

```bash
python3 -m pytest -q
```

手工验证：

- 管理端图像测试页显示当前视觉服务模式。
- 语音测试页显示 ASR 引擎和转写来源。
- 文本测试页命中规则词库关键词时返回非空命中词。

#### 任务 1.2：一键启动、停止和状态检查

目标：提供统一脚本管理四个服务。

当前问题：

- 管理端、视觉服务、文本服务、语音服务需要分别启动。
- 杀旧服务、换端口、查看健康状态依赖手工命令。

实施内容：

1. 新增脚本：
   - `scripts/start_all.sh`
   - `scripts/stop_all.sh`
   - `scripts/status_all.sh`
2. 启动脚本按 profile 加载环境变量。
3. 状态脚本检查：
   - 管理端 `http://127.0.0.1:8000`
   - 视觉服务 `http://127.0.0.1:9000/health`
   - 文本服务 `http://127.0.0.1:8010/health`
   - 语音服务 `http://127.0.0.1:8020/health`
4. 脚本输出每个服务的 PID、端口、健康状态和关键模式。

涉及模块：

- `scripts/`
- `README.md`

验收标准：

- 一条命令能启动完整本地环境。
- 一条命令能停止旧服务。
- 状态命令能指出哪个服务不可用。

#### 任务 1.3：补齐 Docker Compose 编排

目标：让 Compose 覆盖管理端、视觉、文本、语音四个服务。

当前问题：

- `docker-compose.yml` 当前只包含文本和语音服务。
- 管理端和视觉服务仍需要本地手动启动。

实施内容：

1. 为管理端增加 compose service。
2. 为视觉服务增加 compose service。
3. 明确模型目录和存储目录挂载：
   - `./models:/app/models`
   - `./storage:/app/storage`
   - `./data:/app/data`
4. 增加健康检查。

涉及模块：

- `docker-compose.yml`
- 各服务 Dockerfile
- `.env.example`

验收标准：

- `docker compose up` 能启动四个服务。
- 管理端通过环境变量访问三个识别服务。

### 阶段 2：规则与识别质量增强

#### 任务 2.1：统一规则词库发布机制

目标：以管理端规则词库为业务源头，文本和语音服务使用同一份规则快照。

当前问题：

- 管理端 `rule_words` 与服务 JSON 词库存在重复。
- 新增规则后，服务侧规则不一定同步。

实施内容：

1. 增加管理端规则导出接口：
   - `GET /api/rules/export`
2. 增加服务侧规则重载接口：
   - 文本服务：`POST /dictionaries/reload`
   - 语音服务：`POST /dictionaries/reload`
3. 管理端保存规则后，可提示重新发布规则。
4. 规则快照写入 `storage/rules/`，包含版本号和更新时间。

涉及模块：

- `app.py`
- `text_service/routers/dictionaries.py`
- `audio_service/`
- `common/dictionaries/`

验收标准：

- 在管理端新增关键词后，文本测试页能命中新词。
- 同一关键词在文本和语音识别中使用一致的分类和权重。
- 规则快照可回溯版本。

#### 任务 2.2：固化多模态测试样本

目标：将可命中的文本、图片、音频、视频样本固化为可重复测试资产。

当前问题：

- 当前多模态测试内容主要存在运行态数据库。
- 图片和音频命中依赖当前启动模式。

实施内容：

1. 新增目录：
   - `fixtures/text/`
   - `fixtures/images/`
   - `fixtures/audio/`
   - `fixtures/video/`
   - `fixtures/expected/`
2. 提供测试样本清单 `fixtures/README.md`。
3. 新增生成脚本：
   - `scripts/seed_demo_contents.py`
4. 生成以下样例：
   - 纯文本命中
   - 纯图片命中
   - 纯语音命中
   - 图文双命中
   - 视频图像语音命中
   - 白名单无风险内容

涉及模块：

- `fixtures/`
- `scripts/`
- `tests/`

验收标准：

- 清空数据库后可重新导入相同样例。
- 每个样例都有预期风险等级和命中模态。
- 测试脚本可验证文本、图像、语音至少各有一个命中样例。

#### 任务 2.3：真实模型模式校验

目标：确保真实 YOLO 和 Whisper 模式可明确运行、明确失败、明确解释。

当前问题：

- Mock 和真实推理字段已经存在，但缺少端到端验收文档。
- 真实模型首次下载、模型路径、CPU 推理速度等约束未固化。

实施内容：

1. 增加模型检查脚本：
   - `scripts/check_models.sh`
2. 检查：
   - `models/best.pt`
   - `models/enos-smoking-detection-best.pt`
   - Whisper 模型缓存或配置
   - FFmpeg 是否可用
3. README 增加真实模型验收流程。
4. 识别响应中保留：
   - `service_mode`
   - `model_version`
   - `transcript_source`
   - `vision_service_error`
   - `audio_service_error`

验收标准：

- 缺少模型时，真实模式返回明确错误。
- 上传真实音频时，语音服务返回真实转写文本，不使用示例文本。
- 上传测试图片时，前端显示当前模型和权重路径。

### 阶段 3：管理端业务闭环增强

#### 任务 3.1：识别任务状态与失败重试

目标：让识别过程从同步按钮调用升级为可追踪任务。

当前问题：

- 当前识别接口同步执行。
- 失败原因写入部分结果字段，但列表页缺少任务级状态。

实施内容：

1. 增加识别任务表或扩展内容表字段：
   - `task_status`
   - `started_at`
   - `finished_at`
   - `error_message`
   - `retry_count`
2. 前端识别内容列表展示：
   - 未识别
   - 识别中
   - 成功
   - 失败
3. 详情页展示各模态调用状态和错误。
4. 增加失败重试按钮。

验收标准：

- 任意一个服务关闭时，管理端能展示对应模态失败原因。
- 重启服务后点击重试可重新生成结果。
- 失败不影响其他可用模态结果落库。

#### 任务 3.2：证据预览增强

目标：管理端详情页能够直观看到文本、图像、语音证据。

当前问题：

- 文本和图像/语音结果主要以 JSON 展示。
- 证据帧、音频片段和命中词没有形成统一视图。

实施内容：

1. 文本证据展示命中字段、命中词、上下文。
2. 图像证据展示标注图、检测框、OCR 文本、品牌词。
3. 语音证据展示：
   - 转写文本
   - 命中词
   - 时间戳
   - 音频片段播放器
4. 融合结果展示命中模态和贡献分。

验收标准：

- 审核员不需要看原始 JSON 就能判断命中原因。
- 图像、语音命中内容有可见证据。
- 白名单内容能展示降权原因。

#### 任务 3.3：推送载荷规范化

目标：将 Mock 推送升级为结构稳定的监管线索推送模型。

当前问题：

- 推送管理当前偏演示，缺少标准载荷和状态机。

实施内容：

1. 定义推送载荷：
   - 内容基础信息
   - 风险等级和风险分
   - 命中规则
   - 多模态证据
   - 审核记录
   - 操作人和推送时间
2. 推送状态扩展：
   - waiting
   - sending
   - success
   - failed
   - retrying
   - canceled
3. 增加推送请求和响应日志。
4. 保留 Mock 监管平台接口，但结构应与真实接口兼容。

验收标准：

- 推送日志能看到请求摘要、响应摘要和错误原因。
- 失败推送可重试。
- 已确认线索才能进入正式推送队列。

## 5. 验收总标准

完成以上优化后，项目应满足：

1. 可运行
   - 一键启动完整本地系统。
   - 一键停止旧服务。
   - 一键查看四个服务健康状态。

2. 可解释
   - 每个识别结果能说明命中词、模型、证据和风险分来源。
   - Mock/Test/真实模型结果有明确标记。

3. 可复现
   - 测试样本固定。
   - 清空数据库后可重新生成相同演示内容。
   - 自动化测试覆盖文本、图像、语音和融合核心路径。

4. 可闭环
   - 内容可识别。
   - 中高风险可审核。
   - 确认线索可进入推送队列。
   - 推送动作有状态、日志和重试。

5. 可继续生产化
   - 运行配置清晰。
   - 模型和存储路径可配置。
   - 权限、审计、异步队列和部署方案有明确后续任务。

## 6. 风险与约束
